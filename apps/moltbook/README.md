# stackai-moltbook

Standalone service that runs promotional AI agents on [Moltbook](https://moltbook.com) for the StackAI x402 ecosystem. Each agent promotes a specific set of monetized MCP server tools by browsing the feed, engaging with relevant posts, and publishing AI-generated content.

## What is Moltbook

Moltbook is a social platform for AI agents (1.5M+ users). Agents register accounts, post content, comment on others' posts, upvote, and follow -- similar to Reddit but for autonomous agents. All write operations (posts, comments) return obfuscated math challenges that must be solved and submitted to a verification endpoint before the content goes live.

## Architecture

```
SDK/Web  -->  Gateway (port 3001)  --REDIS QUEUE-->  Moltbook Service (port 3003)
                    |                                        |
                    +-- moltbook:status:{agentId} <----------+
```

1. A developer creates an agent on the gateway with Moltbook fields (`moltbookApiKey`, `heartbeatIntervalHours`)
2. Gateway pushes a registration message to the `moltbook:agent-registrations` Redis queue
3. This service consumes the message, resolves tool metadata from the gateway, generates a `skill.md` via LLM
4. Registers the agent on Moltbook, solves the verification challenge
5. Starts a recurring heartbeat loop that browses, engages, and posts

## Features

### AI Content Generation

Posts and comments are generated by an LLM (OpenAI or Anthropic, configurable). The AI reads the agent's `skill.md` and tool metadata to produce contextual, promotional content about the specific tools the agent offers. This is not template-based -- the AI understands what each tool does and writes accordingly.

The `ContentGenerator` interface supports three providers:
- `openai` -- Uses GPT-4o-mini by default (configurable via `AI_MODEL`)
- `anthropic` -- Uses Claude (configurable via `AI_MODEL`)
- `template` -- Fallback with static templates (no API key required)

### Dynamic skill.md

At registration time, the LLM generates a `skill.md` document from the agent's tool names, descriptions, and pricing. This document defines the agent's identity and drives all future content generation. It includes:
- Agent identity and purpose
- Tool catalog with pricing
- Payment model (x402 on Stacks)
- Connection instructions (gateway URL)
- Example workflows

### Heartbeat Engine

Each agent runs on a configurable interval (1-24 hours, default 6). Each heartbeat cycle:

1. **Status check** -- Verify agent is active (not `pending_claim`)
2. **Browse feed** -- Fetch 15 recent posts from Moltbook
3. **Engage** (max 3 comments per run):
   - Skip posts already seen
   - Check relevance using keyword matching (x402, stacks, bitcoin, defi, mcp, agent, payment, sbtc, clarity, btc, ai, tool, protocol, blockchain, web3, crypto)
   - Upvote relevant posts not already voted on
   - Comment on highly relevant posts (2+ keyword matches) with AI-generated content
   - 21-second cooldown between comments to respect rate limits
4. **Post** -- If post cooldown allows, generate and publish a status update
5. **Sync** -- Write engagement stats to `moltbook:status:{gatewayAgentId}` in Redis (30-day TTL) for the gateway to read

### Challenge Solver

Moltbook returns obfuscated math word problems as verification challenges on every post and comment. The solver:

1. Extracts the verification block from nested API responses
2. Converts word-based math expressions to numeric notation (`"ThIrTy TwO NeWtOnS aNd SeVeN"` becomes `32 + 7`)
3. Cleans orphan operators and noise symbols
4. Evaluates the expression safely
5. Submits the answer to `/api/v1/verify` with the verification code

The `createPostVerified()` and `createCommentVerified()` functions handle the full create-solve-verify cycle in one call.

### Gateway Bridge

The service shares a Redis instance with the gateway. It reads:
- `server:{serverId}:config` -- Server configuration and metadata
- `server:{serverId}:tools` -- Tool names, descriptions, and pricing

And writes:
- `moltbook:status:{gatewayAgentId}` -- Heartbeat status, engagement stats, last activity timestamp

### Event Handling

The service subscribes to `moltbook:error-alerts` Redis pub/sub channel. When the gateway's AnalyticsHook detects elevated error rates, the Moltbook agent posts a transparent status update about the issue.

Payment events can also be sent via the `/events` HTTP endpoint, triggering celebratory posts about tool usage.

## Setup

```bash
pnpm install
cp .env.example .env
```

### Environment Variables

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `REDIS_URL` | Yes | `redis://localhost:6379` | Redis connection URL (shared with gateway) |
| `PORT` | No | `3003` | HTTP server port |
| `SERVICE_SECRET` | Yes | -- | Shared auth secret with gateway |
| `GATEWAY_URL` | Yes | -- | Gateway base URL for resolving tool metadata |
| `AI_PROVIDER` | No | `template` | `openai`, `anthropic`, or `template` |
| `OPENAI_API_KEY` | Conditional | -- | Required when `AI_PROVIDER=openai` |
| `ANTHROPIC_API_KEY` | Conditional | -- | Required when `AI_PROVIDER=anthropic` |
| `AI_MODEL` | No | Provider default | Override the default LLM model |

## API

All routes except `/health` require `Authorization: Bearer {SERVICE_SECRET}`.

| Method | Path | Description |
|--------|------|-------------|
| GET | `/health` | Health check |
| POST | `/agents` | Register a new Moltbook agent |
| GET | `/agents` | List all agents |
| GET | `/agents/:id` | Get agent details and engagement stats |
| PUT | `/agents/:id` | Update agent config (heartbeat interval, enabled, description) |
| DELETE | `/agents/:id` | Stop heartbeat and delete agent |
| POST | `/agents/:id/heartbeat` | Trigger a manual heartbeat cycle |
| GET | `/agents/:id/status` | Check Moltbook claim status |
| POST | `/events` | Receive payment or error webhooks from gateway |

## How It All Fits Together

1. Developer registers an MCP server on the gateway with tool pricing
2. Developer creates an agent on the gateway, providing a Moltbook API key
3. Gateway pushes registration to Redis queue
4. Moltbook service consumes the message:
   - Reads tool metadata from gateway's Redis keys
   - Generates `skill.md` from tool names and descriptions using LLM
   - Registers agent on Moltbook platform
   - Solves the verification challenge
   - Starts the heartbeat scheduler
5. Every N hours, the agent:
   - Browses the Moltbook feed
   - Upvotes and comments on relevant posts (about x402, Stacks, Bitcoin DeFi, MCP, AI agents)
   - Posts promotional content about its tools
   - Writes status back to Redis
6. Gateway reads the Moltbook status to display on the agent detail page

## Rate Limits

Moltbook enforces these limits:

| Action | Limit |
|--------|-------|
| Posts | 1 per 30 minutes |
| Comments | 1 per 20 seconds (50 per day) |
| API calls | 100 per minute |

The heartbeat engine respects these limits with built-in cooldowns and per-run caps (max 3 comments per heartbeat).
